Overview

What is Mentoroid?

Mentoroid is a desktop-first coaching and analytics platform designed to help competitive Dota 2 players improve their gameplay through data-driven insights, real-time guidance, and long-term performance tracking. It runs alongside live gameplay, delivering actionable information without interrupting the game experience.

The platform visualizes a player's historical and live match data across multiple dimensions — roles, heroes, game phases, and performance metrics — transforming complex gameplay telemetry into clear, actionable coaching signals. The goal is not just to show statistics, but to tell players what to focus on next and why.

By combining a responsive desktop UI with local-first data handling and an embedded backend, Mentoroid is built to remain stable across long gaming sessions while supporting real-time updates, voice coaching, and post-game analysis flows.

The Challenge

Why is this a hard problem?

01

Real-Time Data in a Desktop-First Context

Unlike web dashboards, Mentoroid must coexist with a full-screen game. The UI must remain responsive even when backgrounded, react to match events in real time, and avoid consuming resources that would degrade gaming performance.

02

Cross-Platform Desktop Complexity

Targeting macOS, Windows, and Linux simultaneously means platform-specific native code, different Steam installation paths, registry queries on Windows, sandboxed environments on macOS, and separate binary compilation targets for the embedded backend.

03

Embedded Backend Orchestration

The AI coaching pipeline requires a companion backend binary to run locally. This binary must be securely distributed, encrypted at rest, decrypted and executed at runtime, and continuously monitored — all within the Flutter app lifecycle.

04

Data Density Without UX Complexity

Dota 2 produces enormous volumes of per-game telemetry. Surfacing meaningful insights — hero performance, role mastery, laning phase vs late game analysis — without overwhelming the player requires careful information hierarchy and visualization design.

05

GSI Integration Across Environments

Game State Integration (GSI) requires writing a configuration file into Dota 2's install directory — which lives in wildly different locations depending on OS, Steam library configuration, and whether macOS sandboxing is active.

Technology Decision

Why Flutter Desktop?

Mentoroid is not a typical analytics dashboard — it is a game-adjacent, real-time desktop application that must coexist with a full-screen game. The framework decision was consequential and non-obvious.

⚡

GPU-Accelerated Rendering

Flutter's Skia/Impeller renderer delivers smooth, consistent frame rates even for graph-heavy dashboards, without relying on a browser engine or heavyweight web stack.

🎯

Single Codebase, Three Platforms

macOS, Windows, and Linux are all served from the same Dart codebase. Platform-specific logic is isolated to native channels, keeping the product surface consistent.

📐

Pixel-Perfect Dashboard Layouts

flutter_screenutil enables a reference-based layout system (1440×900) that scales cleanly across display sizes and DPI settings — critical for data-dense game companion UIs.

🔁

Future-Proof Architecture

The same UI architecture, components, and state management patterns can be reused if Mentoroid expands to mobile or web surfaces in the future, reducing rework cost.

🧩

Rich Ecosystem

Mature packages for charting (fl_chart), routing (go_router), local storage (Hive), and window management gave the team building blocks without reinventing infrastructure.

🕐

Predictable Long-Session Behavior

Flutter's Dart VM is suited for long-running desktop processes, making it reliable during 2–3 hour gaming sessions without GC-induced latency spikes.

Flutter Desktop allowed us to build a high-performance, data-dense UI that behaves like a native desktop app — without splitting into three separate codebases or compromising on visual quality.

Technical Approach

Architecture & Design

Mentoroid follows a layered architecture — the codebase is organized by layer first, with all features sharing each layer. State management is handled through Provider with ChangeNotifier, organized into a dependency-chained hierarchy.

Presentation
screens/ providers/ widgets/
Core
services/ network/ (Dio) config/ analytics/ storage/
Data
repositories/ remote datasources/ local datasources/ models/
Domain
entities/ business logic/

Note on structure: Unlike feature-first Clean Architecture where each feature owns its own data/domain/presentation subtree, Mentoroid groups all features together within each layer — e.g. all providers live in presentation/providers/ and all screens under presentation/screens/. The layered separation is maintained; feature-level isolation is not.

Provider Dependency Tree — providers are chained so that downstream providers automatically receive updated data when upstream state changes:

AuthProvider (root — Steam ID, session, tokens)
├── PerformanceProvider (hero stats, win rates, trends)
│   └── GoalSettingProvider (AI goals, coaching, LLM)
├── GameAnalysisProvider (match history, sessions)
│   └── GoalSettingProvider (shared via ProxyProvider)
├── ProfileProvider (user profile, avatar)
├── GsiSetupProvider (installation detection)
└── PGADetailProvider (post-game analysis details)

Engineering Deep Dives

Challenges & Solutions

Challenge — Embedded Backend Lifecycle

Managing a local backend binary within a Flutter Desktop app is non-trivial.

The AI coaching pipeline requires a companion backend binary to run locally alongside the Flutter app. Distributing, securing, and managing this binary's lifecycle is complex.

Solution — Encrypted Binary Distribution

AES-256-CBC encryption with runtime decryption and subprocess management.

The backend binary is compiled for each target platform and encrypted using AES-256-CBC with a 32-byte environment key. The encrypted binary (backend.enc) is bundled in assets, decrypted at runtime to the application support directory, and launched as a subprocess. A webhook server on port 18100 relays match lifecycle events back to the Flutter layer. File integrity is validated by checking for the binary and its _internal/ dependency folder before each launch.

Challenge — Cross-Platform GSI Configuration

Dota 2's GSI config must be placed in a Steam library path that varies wildly by OS.

Steam can be installed in dozens of locations depending on the OS, user, and whether multiple libraries are configured.

Solution — Multi-Path Detection with VDF Parsing

OS-specific detection strategies with Steam library VDF file parsing.

The GSI service implements OS-specific detection: on Windows it queries the registry and falls back to default paths; on macOS it checks both the sandboxed home and real user home (/Users/$USER); on Linux it checks .steam/steam, .local/share/Steam, and .steam/debian-installation. All strategies parse Steam's libraryfolders.vdf file to discover non-default library locations before writing the config file.

Challenge — Real-Time Game Monitoring Without Drain

The app must detect Dota 2 start/stop without a tight polling loop that hurts in-game performance.

Solution — Adaptive Polling with Process Detection

Dynamic interval switching with platform-native process detection commands.

The GameMonitor service uses an adaptive timer strategy: 3-second polling intervals when the game is not running, switching to 10-second intervals once the game process is detected. Platform-specific commands (pgrep -f dota on macOS/Linux, tasklist on Windows) are used for detection. All timers are cancelled cleanly on app exit to prevent resource leaks.

Challenge — Voice Coaching GSI Sync

Early in development, TTS coaching was silent despite being configured.

The GSI config file was not being written, and the player profile was not being registered with the backend before match start — causing a race condition.

Solution — Startup Orchestration with _ensureGsiConfigured()

A dedicated startup orchestration method that resolves the race condition on app launch.

A startup orchestration method runs on app launch ensuring (a) the GSI config file exists and is valid, and (b) set_player is called against the backend before any match begins. This eliminated the race condition between app startup and game launch that was causing silent coaching sessions.

Challenge — Data-Dense UI Without Visual Overload

Dota 2 generates rich telemetry per match. Presenting all of it simultaneously risks overwhelming players.

Solution — Layered Information Architecture

Progressive disclosure through a layered reveal strategy across UI contexts.

The UI uses a layered reveal strategy: the main dashboard shows an aggregated score and top-line metrics; role and hero breakdowns are surfaced in modals on demand; post-game analysis is a separate screen flow. Interactive fl_chart visualizations with hover tooltips give depth to players who want it, without cluttering the primary view for those who don't.
